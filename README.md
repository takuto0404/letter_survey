# 文字の特徴に関する調査

このリポジトリの `index.html` が GitHub Pages にデプロイされる想定です。ここで行った修正はローカルの Git リポジトリに保存されますが、GitHub のリモートに `git push` するまでは公開ページに反映されません。

## GitHub Pages に反映する手順
1. 変更内容をコミットする（この環境では既にコミット済みになります）。
2. GitHub 上の公開用ブランチ（例: `main` や `gh-pages`）にプッシュする。
3. GitHub Pages の設定で参照ブランチが合っていることを確認する。

`git status` でローカルの変更状況を確認でき、`git log` でコミット履歴を確認できます。GitHub Pages への反映はプッシュ後に数十秒〜数分かかる場合があります。

## 反映確認の目印
- 公開ページでソース表示すると、`APP_VERSION` の定数とページ右下に `build v...` の小さな表示が出ます。ローカルの `index.html` にあるバージョン文字列と一致していれば、更新が反映されています。

## index.html に diff が出ないときの理由
- 直近のコミットで `index.html` を過去のリファレンス版に戻しており、現在のファイル内容がリポジトリの HEAD と完全に一致しているため、`git status` や `git diff` に差分が表示されません。
- 差分を確認したい場合は、`git log -p index.html` で過去のコミットに含まれる変更履歴を確認できます。

## マージコンフリクトが起きたとき
GitHub やエディタが「Accept Incoming Change」などのボタンを出す場合、それぞれの意味は次の通りです。

- **Accept Current Change**: すでに自分のブランチにある変更（`<<<<<< HEAD` 側）を採用する。
- **Accept Incoming Change**: 取り込もうとしている変更（`>>>>>>> branch` 側）を採用する。
- **Accept Both Changes**: 両方の変更を残し、後で手動で整理する。
- **Compare Changes**: どこが違うかを見比べてから選びたいときに使う。

もし誤って「Accept Incoming Change」を押してしまった場合は、以下のいずれかでやり直せます。

1. まだコミットしていなければ、`git checkout --theirs <ファイル名>`（または `--ours`）で片側の変更を取り直すか、`git checkout --merge <ファイル名>` でコンフリクト状態に戻せます。
2. すでにコミットしてしまった場合は、`git reset HEAD^` で直前のコミットを取り消し、ファイルを再度コンフリクト状態に戻してから正しく解消します。

コンフリクトマーカー（`<<<<<<<`, `=======`, `>>>>>>>`）が残っていないか必ず確認し、最終的に意図したコードだけが残るようにしてください。
